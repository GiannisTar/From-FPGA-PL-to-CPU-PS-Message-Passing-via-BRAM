# Simulation & Test Instructions

This file documents how to run the unit and integrated simulations for the PL UART → BRAM design, what each test verifies, and where to find outputs/logs. Save this file for future reference.

Directory
- `project_1/sim/` — all simulation sources, testbenches, and helper scripts live here.

Quick commands
- Run the Icarus (iverilog) multi-test (recommended for fast local verification):

```bash
cd "project_1/sim"
./run_ci.sh
```

- Run a single Icarus simulation manually (compile + run):

```bash
cd "project_1/sim"
iverilog -g2012 -o sim_multi.vvp pl_uart_receiver.v pl_uart_tb_multi.v
vvp sim_multi.vvp
```

- Run the Vivado/XSIM integrated test (on a machine with Vivado installed):

1. Start Vivado and open the Tcl console, or run Vivado in batch mode.
2. In the Vivado Tcl console:

```tcl
cd /absolute/path/to/project_1/sim
source run_xsim.tcl
run_xsim tb_multi     ;# runs the multi-case testbench (tb_multi)
```

What the tests do
- `pl_uart_receiver.v` — behavioral PL-side UART receiver that assembles a 32-bit little-endian length word followed by payload bytes and writes them into a simple memory model `mem[]` (the BRAM model used by testbench).
- `pl_uart_tb.v` — single-case testbench used for focused debugging of single messages.
- `pl_uart_tb_multi.v` — multi-case testbench that runs four test scenarios in sequence:
  1. Zero-length message (length=0). Verifies the DUT writes the length 0 to mem[0..3].
  2. Normal payload. Example: 2-byte payload "Hi". Verifies payload bytes written at `mem[4..]` and length written to mem[0..3].
  3. Truncated payload (declared length > actual sent). Verifies partial payload bytes are written but the DUT does not mark the transfer as completed (length word should not match declared value).
  4. Oversized payload (declared length > capacity). The DUT clamps expected length to the internal maximum payload size and writes up to that many bytes; the test checks for clamp behavior and some sample payload bytes.

Pass/fail behavior
- The multi testbench tracks `pass_count` and `fail_count` and prints a summary line near the end:

  Multi-test complete. PASS=<n> FAIL=<m>

- For Icarus (run via `run_ci.sh`) the testbench uses `$fatal` to terminate with a non-zero simulation exit when `FAIL>0`. The `run_ci.sh` script will therefore return non-zero (and exit with failure) when tests fail.
- For Vivado/XSIM the `run_xsim.tcl` script copies back `sim_xsim.log` and checks for the numeric summary `PASS=<num> FAIL=<num>`; it returns an error if `FAIL>0` (so CI can detect it).

Logs & waveforms
- Icarus (local) outputs:
  - `sim_run.log` — full console output of the `vvp` run when using `run_ci.sh`.
  - `pl_uart_multi.vcd` — VCD waveform generated by the testbench.

- Vivado/XSIM outputs (after `run_xsim tb_multi`):
  - `sim_xsim.log` — the copied simulation log (in `project_1/sim/`).
  - `tb_multi.wdb` — Vivado waveform file (copied back to `project_1/sim/`). Open with Vivado's waveform viewer.

Troubleshooting
- "iverilog: command not found": install Icarus Verilog (e.g., `sudo apt install iverilog` on Ubuntu) or run tests on a machine that has it.
- Vivado errors while running `run_xsim.tcl`: the script copies sources to `/tmp` and runs `xvlog/xelab/xsim` there. Open `/tmp/sim_xsim_<ts>/sim_xsim.log` for detailed vendor tool errors. If the Tcl script reports "Simulation failures detected" then inspect the copied `sim_xsim.log` in this directory.
- If XSIM aborts during elaboration (xvlog/xelab errors), inspect the first error lines in the vendor log; they often show the specific file and line that caused the failure.

CI integration notes
- `run_ci.sh` is intentionally simple: it compiles with Icarus, runs the simulation, writes `sim_run.log`, and exits non-zero if failures occur (either via simulator exit or by scanning for "FAIL" markers).
- `run_xsim.tcl` is safe to invoke inside Vivado Tcl and will return an error if the Vivado simulation log reports any failing tests (via the PASS/FAIL summary or explicit TEST FAIL lines).

Extending tests
- Add additional test scenarios by editing `pl_uart_tb_multi.v` or by adding a new testbench source. Keep the `pass_count`/`fail_count` pattern so the final summary remains machine-readable.

Contact / notes
- If you change the DUT interface (clock/reset names or the memory model), update both testbenches and the Tcl script copy list (`run_xsim.tcl`) accordingly.

Building for hardware
- To synthesize/implement and export an XSA for Vitis, use the build scripts in `project_1/scripts`:

```bash
cd project_1/scripts
./build_bitstream.sh    # runs Vivado batch build and exports pl_ps_bd_wrapper.xsa
```

Note: Vivado must be installed on the machine and available in PATH. These scripts only generate the build in batch mode; inspect Vivado logs if any step fails.
